## 二、通用规范（全站，参考爱搜题题目列表页视觉）

### 2.0 设计风格（全站基调，任何页都不得偏离）

- **UI 框架**: **Vben Admin 5.7 + Naive UI**（web-naive app 壳）
  - Vben 提供：布局骨架（顶栏/侧栏/标签页/面包屑/折叠）、权限拦截、i18n、主题、抽屉/弹窗封装、表格/表单封装（`VbenTable` / `VbenForm`）、基础组件 `@vben/common-ui`
  - Naive UI 提供：数据组件（`n-data-table` / `n-tree` / `n-select` / `n-date-picker` / `n-tag` / `n-modal` / `n-drawer` / `n-popconfirm` / `n-alert` / `n-empty` / `n-steps` / `n-tabs` / `n-progress` / `n-code` / `n-switch` / `n-radio-group` / `n-checkbox-group` / `n-pagination` 等）

- **选型硬约束**:
  - **禁止混入其它 UI 库**（不引 Ant Design / Element Plus / TDesign，避免视觉割裂 + 依赖膨胀）
  - 所有表格走 Naive `n-data-table` 或 Vben 封装 `VbenTable`（不自己造 `<table>`）
  - 所有弹窗走 `n-modal`（居中）或 `n-drawer`（抽屉，≥6 字段或含长文本时用抽屉）
  - 所有表单走 `VbenForm`（`@vben/common-ui`）+ `z`(zod) 校验 schema；不写纯手工 `<form>`
  - 图表走 `vue-echarts`（或 Naive 未覆盖时引 echarts），不引 Chart.js / AntV 系列
  - Icon 走 `@iconify/vue` + Naive 官方 `NIcon` 封装；禁止直接贴 SVG 字符串

- **视觉基调**（Vben 默认 + Naive 默认色板，**不做自定义覆盖**）:
  - 主色 `#2080f0`（Naive 默认蓝）；红 `#d03050`；绿 `#18a058`；橙 `#f0a020`；灰阶 `#333 / #666 / #999 / #eee / #f5f6f8`
  - 圆角 4-8px；投影极浅（Naive 默认）；无渐变、无拟物，扁平
  - 表格密度：默认 `medium`，工具栏可切换 `small / large`（`VbenTable` 内置）
  - 字号：正文 14px，表格 13px，标题 h4=18px / h3=20px / h2=24px；数字类大号（价格/指标卡）28px 加粗

- **交互基调**:
  - 按钮主色只给"主要动作"（新建/保存/下一步），其它用 `quaternary` 或 `default` 次按钮
  - 危险操作（删除/禁用/强制下线）统一 `n-popconfirm` 或 `n-modal` 二确，并用红色次按钮
  - 加载状态：表格 skeleton，按钮自带 `loading` 属性；**不写**手工 spinner
  - 通知：页面级错误 `n-alert`（常驻），瞬时反馈 `message.success/error/warning`（≤1.5s），重要告警 `n-notification`（右上角，可关）

- **页面脚手架落地**:
  - 所有页面放在 `apps/web-naive/src/views/{user|admin}/<page>/index.vue`
  - 公共组件放 `apps/web-naive/src/components/`（跨页复用）或 `packages/@core/ui-kit/`（跨 app 复用，本仓只有 naive 一个 app，一般不用）
  - 页面骨架优先用 Vben 的 `<Page>` 容器（自带 padding / 标题栏）

### 2.1 布局

- **顶栏**（h=48）：左侧 `logo` → `折叠按钮` `刷新` `首页 icon` `面包屑`；右侧 `全屏` `通知铃铛(未读数)` `头像+昵称` `主题切换` `菜单`。
- **左侧栏**（w=220 展开 / 64 折叠）：浅色底、当前项蓝底高亮、组带展开箭头；底部常驻折叠按钮。
- **主区**：顶部 `Tab 栏`（首页 tab 固定不可关）+ 下方 `内容区`。
- **内容区结构**：`筛选卡片`（灰底） → `操作栏`（左业务按钮 / 右工具图标） → `表格` → `分页`。

### 2.2 分页

- 位置：表格底部右侧。
- 组件：Naive `n-pagination`，展示 `总数 / 每页大小切换(10/20/50/100) / 页码 / 快速跳转`。
- 列表接口统一 query：`page` `page_size`，响应 `{list, total, page, page_size}`。

### 2.3 表格

- **操作栏**：
  - 左：业务按钮 `+新建`(主色) / `删除`(红, 依赖选中) / `导出`(白底 + icon)。
  - 右：工具图标区 `刷新` `下载` `打印` `密度` `列设置` `全屏`（全部走 `n-data-table` 内置或 Vben `VbenTable` 的 toolbar 插槽）。
- **复选框列**：最左，`row-key=id`，批量选中后顶部红色"删除"按钮变可用并显示数量。
- **序号列**：`#` 自增（跨页连续）。
- **状态/来源/类型**：彩色 `n-tag`（成功=绿 / 进行中=蓝 / 失败=红 / 默认=灰）。题型等二级枚举用 `text-link` 风格（下划线+主色）。
- **操作列**：文字链接风格，竖线 `|` 分隔；蓝=修改/查看/详情，红=删除/停用。
- **空态**：`n-empty` + 中心描述 + (可选)"新建"主操作按钮。

### 2.4 表单

- **筛选表单**（表格上方卡片）：
  - 默认展开两行、每行 4 列（`label:input` 水平排列）。
  - 按钮区：`查询`(主色) + `重置` + `收起^/展开v`（文字链接，切换显示更多筛选）。
  - 日期范围用 `n-date-picker type=daterange`；下拉用 `n-select` 远程搜索。
- **新建/编辑**：
  - 字段少(≤6)：`n-modal` 居中弹窗。
  - 字段多/含富文本：右侧 `n-drawer`（宽 720）。
  - 校验：`@vben/common-ui` 的 `z` (zod) schema；失败不关闭弹窗。

### 2.5 错误态 / 空态 / Loading

- 表格：请求中 skeleton；失败时保留上次数据 + 右上角红色"加载失败"tag + 重试按钮。
- 空数据：`n-empty` + 描述 + 主操作。
- 全局 `40002`（token 过期）：清 store 跳 `/auth/login?redirect=xxx`。
- 全局 `40003`（无权）：按钮隐藏 + 整页跳 `/_core/fallback/forbidden`。
- 全局 `40006`（配额不足）：页面内 `n-alert` 警告 + 顶部气泡。
- 全局 `50001`（下游故障）：页面顶 `n-alert type=error` 常驻红 banner。

### 2.6 权限降级

- `accessMode='backend'`，菜单由 `GET /auth/menus` 返，前端不做菜单过滤。
- 按钮级权限：`<span v-if="hasAccess(['question.manage'])">` 或 Vben 内置 `<AccessControl :codes="[...]">`。
- 无权访问具体路由：跳 `/_core/fallback/forbidden`，提示"联系管理员到 /admin/role 分配"。

### 2.7 i18n

- 默认 `zh-CN`，`en-US` 搭了框架但不强制完整；新增 key 两语都写。
- 文案文件：`apps/web-naive/src/locales/langs/{locale}/{module}.json`，`module` 与页面组对齐（`user.json` / `admin.json` / `page.json` / `common.json`）。
- 纯业务词（题型、来源、状态枚举）放对应 `module.json` 的 `enum.xxx` 命名空间。

#### 3.1.4 在线充值 — `/user/recharge`

- **权限**: `portal.access`
- **页面目标**: 用户自助给账户钱包充值（非套餐购买；套餐购买走 3.1.3 我的套餐）；三步向导式，解耦下单-支付-回执
- **布局类型**: 单卡全宽 + 顶部 3 步 Steps 向导；无左右分栏
- **数据来源**:
  - `POST /user/order/create` — 🆕 需支持 `order_type=recharge`（区别于套餐购买 `order_type=plan`），入参 `{order_type, amount, pay_method}`，出参 `{order_id, pay_url, qr_code_url, out_trade_no}`
  - `GET /user/order/list?order_id=xxx` — 轮询支付结果（已有，过滤 `status` 字段）
  - `GET /admin/payment-config/list`（**公开只读版** 🆕 `/user/payment-methods`）— 返启用中的支付渠道 `[{code:'alipay', name:'支付宝', icon:'...', enabled:true}, ...]`，前端按此渲染支付方式按钮组（管理端开关哪个就展示哪个）

- **Steps 组件**（Naive `n-steps`，顶部居中，3 步）:
  - `1. 第一步 / 填写充值信息` — 默认激活
  - `2. 第二步 / 扫码支付`
  - `3. 第三步 / 支付结果`
  - 步骤条走完不可跳跃回退（`clickable=false`）；点「上一步」可手动回退到第一步（第二步超时时）

- **Step 1 · 填写充值信息**（水平 label-right Form，label 宽 80，居中 800px 容器）:
  - **字段 1**: `充值金额` — `n-input-number`，`min=1` `max=system_configs.recharge_max`（默认 10000）`precision=2`；右侧灰色小字 `元`；placeholder `请输入金额`
    - 可选：下方 5 个快捷面额按钮（`10 / 50 / 100 / 200 / 500`），点击直填入（从 `system_configs.recharge_quick_amounts` 拉取，逗号分隔；空则不展示）🆕
  - **字段 2**: `支付方式` — `n-radio-group` 样式为按钮组（`type=button`），每个按钮左侧 icon + 文字：
    - `支付宝` — 支付宝蓝 icon
    - `微信` — 微信绿 icon
    - `QQ钱包` — QQ 企鹅 icon
    - 列表由 `/user/payment-methods` 动态渲染；无启用渠道时展示 `<n-alert>` 红条 `暂无可用支付方式，请联系管理员`
  - **按钮组**（字段下方居左，label 列偏移）:
    - `下一步` — 蓝底白字主按钮，提交前校验 `金额>0 && 支付方式非空`；点击 → 调 `POST /user/order/create` → 成功拿 `{order_id, qr_code_url}` → 进入 Step 2
    - `重置` — 白底灰边次按钮，清空 form 回初始态
  - **字段校验**:
    - 金额 `< min`：红字 `金额不能低于 {min} 元`
    - 金额 `> max`：红字 `单笔最多 {max} 元，如需大额充值请联系客服`
    - 非数字/负数：zod 层拦截

- **Step 2 · 扫码支付**:
  - 居中展示：
    - 二维码图（由 `qr_code_url` 渲染或直用后端返的 base64 `data:image/png;base64,...`，240×240）
    - 二维码下方：`请使用 {pay_method_name} 扫码支付`（灰色 14px）
    - 金额大字 `¥ {amount}` 红色 36px
    - 订单号 `订单号: {out_trade_no}`（灰色 12px，右侧复制图标）
  - **轮询**: 每 3s 调 `GET /user/order/list?order_id=xxx`，命中 `status=success` → Step 3；`status=failed/cancelled` → Step 3（失败态）；`status=pending` → 继续轮
  - **超时**: 10 分钟未支付成功，停止轮询 + 显 `n-alert` 橙 `订单已超时，请重新下单` + 展示「重新下单」按钮跳回 Step 1
  - **按钮**: 底部 `上一步`（跳回 Step 1，订单仍保留 pending 状态，用户可以再下一单）；次按钮 `已支付完成，手动刷新`（触发一次轮询，避免网络延迟导致界面卡在二维码）
  - **离开保护**: `onBeforeRouteLeave` 若仍在 pending 状态，弹 `n-modal` 二确 `订单尚未完成支付，离开后需要重新下单，确定离开？`

- **Step 3 · 支付结果**:
  - **成功态**（status=success）:
    - 居中大绿勾 icon（128×128）
    - `充值成功`（h3 黑字加粗）
    - 明细两行：`充值金额 ¥ {amount}` / `订单号 {out_trade_no}`
    - 按钮组：`返回首页`（主按钮，跳 `/user/dashboard`）/ `查看余额日志`（次按钮，跳 `/user/log/balance`）
  - **失败态**（status=failed/cancelled/timeout）:
    - 居中大红叉 icon
    - `充值失败`（h3 黑字）
    - 失败原因（从订单详情 `fail_reason` 取，无则展示 `支付未完成`）
    - 按钮组：`重新下单`（主按钮，回 Step 1）/ `返回首页`（次）

- **交互细节**:
  - 三步状态由 ref 驱动（不走路由）；刷新页面即回 Step 1
  - 从 3.1.3 「立即购买」跳进来时带 `?plan_id=xxx` → Step 1 的金额/支付方式字段**置空**（因为套餐购买走专门接口，不是 recharge）或直接跳转到 Step 2（已下单）——需后端确认路由语义
  - 金额输入支持键盘 Enter 提交
  - 二维码过期视觉提示：二维码上叠加半透明灰层 + 刷新 icon 按钮（点击回 Step 1）

- **边界 & 异常**:
  - `/user/order/create` 返 40006：弹 `n-alert` `当前有未完成订单，请先处理`（后端幂等防刷）+ 跳转查看现有订单
  - 轮询期间 50001：暂停轮询 + 显 `服务异常，稍后自动恢复`（每 30s 尝试恢复）
  - 用户在 Step 2 主动关页：订单留 pending 态，后端靠支付平台异步 callback 最终决断
  - 支付渠道全部禁用：Step 1 的「下一步」灰掉 + tooltip `无可用支付渠道`

- **埋点**: 每步 Step 进入 → `operate_logs`（module=recharge, type=step_{1|2|3}）；成功 → `type=recharge_success`；失败 → `type=recharge_failed` with `fail_reason`

- **待确认**:
  - `POST /user/order/create` 是否已支持 `order_type=recharge`？目前似乎只走套餐单 🆕
  - 公开的支付渠道枚举接口 `/user/payment-methods` 是否新建？🆕
  - 支付成功回 Step 3 的**权威来源**：前端轮询 vs 第三方同步跳转回 `/callback/epay/return`？建议：两路都走，以后端 `/user/order/list` 为权威

- **按钮组**（表格工具栏左侧）:
  - `+ 添加` — 蓝色主按钮 → 打开编辑 Modal（新建态）
  - `删除` — 红色次按钮 → 批量删除勾选行（≥1 行勾选才亮起，否则灰）→ 二确 Modal `确定删除选中的 N 条接口？`
- **工具栏右侧**（标准）: 刷新 / 导出 / 打印 / 紧凑 / 列设置 / 全屏

- **表格列**:
  - ☐ — 复选框（支持批量）
  - `接口名称` — 文本，hover 显完整名；超长截断
  - `接口地址` — monospace，灰色小字；长 URL 截断显前 40 + `...`，右侧复制 icon
  - `请求方式` — tag：`GET=蓝 / POST=紫`
  - `关键词参数` — 文本（即第三方接口里代表"搜索词"的 param key，通常是 `q` / `title` / `question`）
  - `类型选项参数` — 文本（即第三方接口里代表"题型+选项"的 param key，通常是 `info` / `options`）
  - `超时(秒)` — 数字，右对齐；默认 3；>10 红字告警
  - `排序` — 数字（整数，越小越优先轮询）
  - `状态` — `n-switch`（开/关，直接切换不进 modal），关闭态后端不再轮询此源
  - `创建时间` — `YYYY-MM-DD HH:mm`
  - `操作`（固定右，3 按钮）:
    - `测试` → `POST /user/api-source/test`，行末 loading spinner → 返回后弹 `n-popover` 或 `message` 展示连通性（HTTP 状态码/响应耗时/提取到的答案预览）
    - `编辑` → 打开编辑 Modal（编辑态，预填 detail 数据）
    - `删除` → `n-popconfirm` 悬浮二确，确定后 `DELETE /user/api-source/delete?id=xxx`

- **C. 「添加 / 编辑」Modal**（`n-modal` w=640，Form label-top）:
  - **字段组 1 · 基本信息**:
    - `接口名称` — `n-input` 必填，最长 32；唯一性由后端约束，重复时返 `40004 接口名称已存在`
    - `接口地址` — `n-input` 必填，zod URL 校验，仅允许 `http://` `https://`；trim 首尾空格
    - `请求方式` — `n-radio-group` 按钮组 `[GET / POST]`，默认 GET
    - `请求头（可选）` — `n-dynamic-input` 动态键值对（header name + value），最多 5 对；可放自定义鉴权头 `Authorization: Bearer xxx`
  - **字段组 2 · 参数映射**（告诉后端"第三方接口里哪个字段是关键词/选项"）:
    - `关键词参数` — 必填，单行输入，placeholder `第三方接口里代表"搜题关键字"的字段名，如 q / title / question`
    - `类型选项参数` — 选填，placeholder `第三方接口里代表"题型+选项"的字段名，如 info / options（留空则不传）`
    - `附加固定参数（可选）` — `n-dynamic-input` 动态键值对，用于补接口的鉴权 key / appid 等固定参数（如 `token=xxx`）
  - **字段组 3 · 解析配置**（告诉后端"从第三方响应里挖答案"）:
    - `响应路径 (JSONPath)` — 必填，placeholder `如 data.answer / result.content / $.data` 默认 `data`
    - `成功码字段` — 选填，`n-input`，默认 `code`
    - `成功码值` — 选填，`n-input`，默认 `1`；与字段配对用于判定"第三方接口是否业务成功"
  - **字段组 4 · 运行控制**:
    - `超时(秒)` — `n-input-number`，min=1 max=30，默认 3
    - `排序` — `n-input-number`，min=0 max=999，默认 0（越小越优先）
    - `状态` — `n-switch`，默认开
  - **底部按钮**: `取消` / `测试接口`（不提交，先调 `/test` 回显） / `保存`
  - **「测试接口」行为**: 用当前表单值调 `POST /user/api-source/test?dry_run=1`，返回 `{status_code, response_ms, sample_answer, raw_response_preview}` → 以 `n-alert` 内嵌结果（绿=成功，橙=超时，红=解析失败+展示 raw 片段）；测试不入库

- **D. 连通性测试结果（表格行「测试」按钮）**:
  - 成功：`n-popover` 绿框，展示 `HTTP 200 / 耗时 {ms}ms / 预览答案: "xxx"`
  - 失败：`n-popover` 红框，展示 `HTTP {code} / 原因 {err}`，按钮 `查看原始响应` 打开 `n-modal` 大窗贴 raw JSON
  - 超时：橙框 `> {timeout}s 超时`，提示 "可尝试提高超时或检查网络"

- **交互细节**:
  - `n-switch` 状态切换直接触发 `PUT /update`（只改 status 字段），成功后原地回显，失败回滚 switch 并 message.error
  - 保存前前端做 URL 合法性 + 必填校验；后端再做一次深校验（URL host 不在本站黑名单、接口地址非内网等）
  - 「测试接口」按钮限流前端 3s 一次；后端限流由 `/test` 接口自身保底
  - 删除二确里若接口当前 `status=1`，文案加红字强调 `此接口当前启用中，删除后搜题将不再轮询此源`

- **边界 & 异常**:
  - 空态：表格展示 `<n-empty>` + 空盒图（截图样式） + 提示文字 `还没有配置题库源。点击上方"+添加"挂入第三方接口`
  - 40003：当前用户没有配置题库源的权限（RBAC 控制，本期 `portal.access` 覆盖，短期不会触发）
  - 40004 `接口名称已存在`：Modal 内字段下方红字提示
  - `/test` 50001：接口返红 alert `测试服务暂不可用，请稍后`
  - 用户配置的第三方 URL 本身 5xx：前端测试展示真实错误，**不**算平台 50001

- **埋点**: 新增/编辑/删除/切换状态/测试 → `operate_logs`（module=api_source, type=create/update/delete/toggle/test）

- **已决策**:
  - 无独立筛选卡（列表规模小，通常用户配置的题库源 < 20）；若后期量大再加
  - 删除走**物理删除**（这张表跟业务数据无严格外键关系）；如要软删改 `DELETE` 语义需后端配合
  - 状态 switch **原地切换**，不弹确认（低风险 + 提高体验）
  - 「关键词参数」和「类型选项参数」的命名保持与截图一致，便于用户对照 3.1.5 的接口文档模板理解"映射"概念

- **待确认**:
  - 用户端配置的题库源上限？建议 `system_configs.user_api_source_limit`（默认 10），超过限制 Modal 内保存按钮置灰
  - 配置的 URL host 白/黑名单策略？（建议禁止 `127.0.0.1 / localhost / 内网段`，后端强校验）🆕
  - 附加固定参数里用户可能想放"鉴权 token"，这是明文存储的敏感字段，表 `api_sources.extra_params` 需加密落盘还是原文？建议**不加密**（用户自填自用，不构成服务方责任），但字段值在列表页**不展示**（只在编辑 Modal 展示）

- **页面目标**: 展示管理端下发的公告（产品更新、维护停机、活动通知），用户**只读**查看，不能评论/点赞/转发
- **布局类型**: 筛选卡（一行）+ 表格 + 详情 Drawer（与 3.1.10 同模板）
- **数据来源**:
  - `GET /user/announcement/list?title&creator&type&page&page_size` — 列表（已有）
  - `GET /user/announcement/detail?id=xxx` — 详情（已有，返回富文本 Markdown/HTML + 元信息）

- **筛选条件**（一行，全量展开，无折叠；3 字段）:
  - `公告标题` — `n-input`，模糊匹配；placeholder `请输入`
  - `创建者` — `n-input`，匹配发布者用户名；placeholder `请输入`
  - `公告类型` — `n-select`，选项 `[公告 / 通知 / 活动 / 维护]`（由 `announcements.type` 枚举驱动），placeholder `请选择`；单选
- **按钮**: `搜索`（蓝主按钮，icon + 文字） / `重置`（次按钮）

- **表格**（无顶部左侧按钮组，只保留右侧工具栏）:
  - 工具栏右侧: 刷新 / 导出 / 打印 / 紧凑 / 列设置 / 全屏

- **表格列**:
  - ☐ — 复选框（批量操作本期**不做**，保留列为扩展；用户端日志只读原则对公告同样适用）
  - `#` — 序号，从 1 起，按分页计算
  - `公告标题` — 文本，蓝色链接态，点击 = 打开详情 Drawer；未读公告在标题左侧加红点 dot（`unread=true` 由后端根据 `announcement_reads` 关系表返）
  - `公告类型` — tag（`公告=绿 / 通知=蓝 / 活动=紫 / 维护=橙`）
  - `状态` — tag：`正常=蓝`（后端 `status=1`）/ `已下线=灰`（`status=0`）；正常下线态用户端正常不会看到，除非用户曾读过已归档的公告
  - `创建者` — 文本，显示发布者用户名（通常是 `admin`）
  - `创建时间` — `YYYY-MM-DD HH:mm:ss`
  - `操作`（固定右）:
    - `👁 详情` — 蓝字链接，点击打开详情 Drawer
    - （**不**提供删除/隐藏/标记已读按钮——"已读"状态由打开详情 Drawer 时后端自动标记，前端无按钮）

- **分页**（底部居中，截图样式）:
  - 文案 `共 {total} 条` + `n-pagination` + `{page_size}条/页` 下拉 + `前往 [ ] 页` 跳转
  - 默认 `page_size=10`，可选 `[10 / 20 / 50]`
  - 与全站通用分页 2.2 规范一致

- **详情 Drawer**（右侧，w=720）:
  - 顶部元信息区（灰底，20px 内边距）:
    - 标题（h3 加粗，两行可换行）
    - 第二行小字：公告类型 tag / 创建者 / 创建时间 / 阅读数（若后端返）
  - 内容区：Markdown 渲染（`n-code` 代码块支持、`n-a` 链接、图片自动懒加载）；若后端返 HTML 则做 XSS sanitize 后渲染
  - 底部：`关闭` 按钮
  - **打开时行为**: 调 `/user/announcement/detail?id=xxx` → 后端隐式把 `announcement_reads` 插一行（用户 × 公告 的已读关系），前端收到成功响应后**同步把当前行 `unread` 改为 false**（红点消失）

- **交互细节**:
  - 首次进站/登录后：若 `list?unread=1&limit=1` 有未读重要公告（`type=维护` 或管理端勾了 `is_pinned`），在 `/user/dashboard` 顶部弹 `n-alert` 横幅（见 3.1.1）；本页不弹，避免重复打扰
  - 默认排序：`is_pinned desc, created_at desc`（置顶公告永远在前）
  - 置顶公告在标题列加金色 📌 icon 前缀
  - 行 `hover` 高亮（全站表格通用）；点击整行**不**触发详情，避免误操作 —— 必须点详情链接或标题链接

- **边界 & 异常**:
  - 列表为空：`<n-empty>` `暂无公告`
  - 详情 404（公告被删）：Drawer 内展示 `<n-result status='404'>此公告已删除`
  - 50001：顶部红 banner，表格骨架保持
  - 未读数：若后端不返 `unread` 字段，前端不展示红点（降级方案）

- **埋点**: 打开详情 → `operate_logs`（module=announcement, type=view_announcement, extra={id, title}）

- **已决策**:
  - 用户端公告**只读**（无导出、无删除、无评论、无订阅）
  - 已读状态由打开详情**自动标记**，无「标记已读」按钮
  - 公告内容走 Markdown/HTML 二选一由管理端 3.2.9 决定（建议 Markdown，无富文本编辑器负担）
  - 行操作列只保留 `详情` 一个动作，视觉上与其它用户端日志页（3.1.10 搜题日志等）对齐

  - 0021 已完成的四个分组父节点（`/admin/group/*`）的 `component` 字段应为空（分组不渲染页面，只做分组容器）—— 前端路由层要对 `component=null` 的节点特殊处理，不注册实际路由

  - `+ 新增代理` — 单条 Modal（字段：协议/IP/port/账号/密码/地区/备注）
  - `⚡ 快速添加` — `POST /quick-add`：弹 Modal 贴 `ip:port:user:pass` 单行（本期用 🆕）
  - `📥 批量导入` — 弹 Modal 上传 CSV / 贴多行文本（每行一条 `protocol://user:pass@ip:port`），解析后预览确认再 `POST /batch-import`
  - `📤 批量导出` — `GET /batch-export` 下载 CSV
  - `🔍 全量探测` — `POST /probe-all` 触发后台任务；展示进度（轮询后端返的 `{probed, total}` 百分比）
  - `删除` — 批量删除（勾选 ≥1）

- **表格列**:
  - ☐ / `ID` / `协议` tag / `IP:Port` monospace / `账号` 脱敏 / `地区` / `延迟(ms)` 数字 / `成功率 24h` / `状态` switch / `健康` dot+tag / `最近探测` / `创建时间` / `操作`
  - 延迟：<200ms 绿 / 200-800 橙 / >800 红 / 无数据 灰
  - 成功率：<50% 红字

- **行操作**: `探测`（单次 probe，行末 loading + 结果 popover） / `编辑` / `禁用/启用` / `删除`

- **批量导入 Modal**（核心）:
  - 两种模式 Tab:
    - **Tab 1 上传 CSV**: 下载模板按钮 + `n-upload` 拖放区；上传后后端返校验结果（合法 N 条 / 重复 M 条 / 格式错 K 条），展示表格预览，确认后 `POST /batch-import`
    - **Tab 2 贴文本**: textarea（每行一条），解析格式同 quick-add；实时 client-side 校验（高亮错误行）
  - 底部按钮: `取消` / `导入`

- **交互细节**:
  - 「全量探测」是异步任务（代理多时可能几分钟），提交后顶部出现悬浮进度条 `探测中 23/150`，完成后 message.success；其它操作不阻塞
  - 密码字段默认 mask；编辑 Modal 内可见明文（管理员视角）
  - 探测结果 popover 显示 `HTTP {code} / 延迟 {ms}ms / 出口 IP {detected_ip}`

- **边界 & 异常**:
  - 批量导入：合法 + 重复 + 格式错并存时，只导入合法条，返回报告（提示 `成功 N / 重复 M 忽略 / 错误 K`）
  - 全量探测触发期间再次点击：后端返 `已有探测任务运行中` → 前端 toast

- **埋点**: 批量导入 / 全量探测 / 禁用 → `operate_logs`（导入记 `imported_count`，不记具体 IP 列表）

- **已决策**:
  - 代理 IP 仅限后台管理员维护，用户不接触
  - 探测结果健康度**软记忆**（持续 3 次失败才判定"异常"），避免单次抖动误判
  - 导出 CSV 的"密码"列**不脱敏**（导出文件需要能重新导入）—— 下载前二确 + 审计日志

#### 3.2.20 采集任务 — `/admin/collect`

- **权限**: `collect.manage`
- **页面目标**: 管理员查看**全站**采集任务，对失败任务重试/强制停止；与用户端 3.1.6 的区别 —— 跨用户视图 + 可管控
- **布局类型**: 筛选卡 + 表格 + 详情 Drawer（与 3.1.6 同模板，多一列"用户"）
- **数据来源**:
  - `GET /admin/collect/task/list?user_id&account&platform&status&date_from&date_to&page&page_size`
  - `GET /admin/collect/task/detail?id=xxx`
  - `POST /admin/collect/task/stop` — 强制停止运行中的任务
  - `POST /admin/collect/task/retry` — 重试任务

- **筛选**（比 3.1.6 多 `用户`）:
  - `用户` — `n-select` 搜用户名
  - `账号` / `采集类型` / `状态` / `创建时间`（同用户端）
  - 展开：`课程数范围` / `题目数范围`（数字区间筛选，发现"只抓到 0 题" 的异常任务）

- **按钮组**:
  - `批量重试`（勾选 ≥1 的失败任务） / `批量停止`（运行中） / `导出` / `清理 30 天前成功任务`（🆕 可选）

- **表格列**（与 3.1.6 对齐 + 多 3 列）:
  - ☐ / `ID` / `用户` / `账号` 脱敏 / `采集类型` tag / `课程数` / `题目数` / `状态` / `错误信息` / `耗时(秒)` / `worker 节点` 🆕 / `创建时间` / `操作`

- **行操作**: `详情` / `重试`（失败/部分成功）/ `强制停止`（运行中，二确 + reason 必填）/ `查看 worker 日志`（跳到 3.2.13 监控页锚定到对应 worker）

- **详情 Drawer**: 复用 3.1.6 D 段，额外显示:
  - 用户基本信息
  - worker 执行节点、PID、日志片段
  - 管理员行动记录：谁在什么时候做了 `重试/停止` 操作

- **交互细节**:
  - 大批量重试前二确：`将重试 N 个任务，可能引发第三方平台风控，建议分批，确定？`
  - 强制停止需填 reason 写入 `operate_logs`
  - 表格行状态 dot 与 3.1.6 对齐配色

- **边界 & 异常**:
  - 强制停止后 worker 短暂无法响应：状态先置 `stopping`，后端回调确认后转 `cancelled`
  - 批量重试触发第三方平台风控：后端建议串行 + 间隔

- **埋点**: 重试 / 停止 / 批量操作 / 查看详情 → `operate_logs`

- **已决策**:
  - 管理端可看所有用户任务（含账号脱敏，hover 明文，明文访问记审计）
  - 强制停止必填 reason
  - 批量重试加二确 + 建议分批

#### 3.2.21 采集配置 — `/admin/collect-config`

- **权限**: `collect_config.manage`（或归入 `collect.manage`）
- **页面目标**: 调整采集引擎的**全局参数**（并发数、延迟、重试、代理策略、UA 池、各平台的抓取脚本版本等），不影响具体任务的运行态
- **布局类型**: 左侧分组 Tab（vertical） + 右侧配置表单；与 3.2.6 系统配置类似
- **数据来源**:
  - `GET /admin/collect-config/list?group=xxx`
  - `POST /admin/collect-config/update`

- **A. 左侧分组 Tab**:
  - `并发与限速 concurrency` — 全局 worker 并发数 / 每平台并发上限 / 单账号间隔秒数
  - `重试策略 retry` — 最大重试次数 / 重试间隔（等差 or 指数退避）/ 遇哪些错误不重试（枚举：密码错误 / 账号冻结 等）
  - `代理 proxy` — 是否启用代理池 / 代理类型优先级 / 单代理最大并发
  - `UA 池 user_agent` — JSON 数组，每行一条 UA 字符串；右侧按钮「校验 UA 有效性」抽样请求
  - `平台脚本 platform` — 每个平台（超星/智慧树/学习通/知到）一段可折叠 section：
    - 脚本版本号（只读）
    - 登录成功关键字、失败关键字（正则）
    - 是否启用二次验证（滑块/短信）
    - 课程列表页 URL 模板 / 题目抓取接口
  - `风控 risk_control` — 触发风控关键词列表 / 触发后的静默时长 / 告警阈值

- **B. 右侧表单**:
  - 与 3.2.6 共享字段渲染器（动态表单），不同 `value_type` 对应控件
  - 顶部显示当前分组的 `require_restart` 聚合警告（若有任一字段需重启）
  - 底部: `保存` / `重置为默认` / `导出当前分组配置为 JSON`

- **C. 「测试平台脚本」按钮**（每个平台 section 右上角）:
  - 点击弹 Modal，输入测试账号 → 后端 dry-run 一次登录流程，返回各步骤成功/失败（用于脚本版本升级后的回归测试）🆕

- **交互细节**:
  - 改动生效策略：
    - `并发 / 重试 / 代理` — 立即生效（新任务按新配置跑，老任务不受影响）
    - `脚本版本` — 需重启 worker 生效，保存后弹提示
  - 高危字段（"禁用风控检测"之类）二确加红字

- **边界 & 异常**:
  - JSON 数组字段（UA 池）格式错：保存按钮灰
  - 并发数 > 后端设定上限：字段下方红字提示 `超出系统承载上限 {max}`

- **埋点**: 每字段改动 → `operate_logs`（module=collect_config）

- **已决策**:
  - 与 3.2.6 系统配置拆开是因为：3.2.6 面向**平台经营**（价格/注册开关/接口地址），本页面向**采集运维**（并发/代理/UA/脚本），受众不同、变更节奏不同
  - 脚本版本号只读展示，升级通过 CLI 部署（避免 UI 改错致全平台瘫痪）
  - 平台脚本测试按钮走 dry-run，不持久化数据

- **待确认**:
  - 脚本测试功能 `dry-run` 是否已有后端实现？🆕
  - UA 池是否需要与代理池配对使用（某代理固定搭配某 UA）？建议本期不做，后续按"指纹"维度再建单独表
